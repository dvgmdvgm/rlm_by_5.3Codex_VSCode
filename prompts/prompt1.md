# ТЕХНИЧЕСКАЯ СПЕЦИФИКАЦИЯ (ТЗ): Разработка MCP-сервера для Гибридной Системы Рекурсивной Памяти (Hybrid RLM Memory System)

## 1. Цель проекта
Необходимо разработать MCP (Model Context Protocol) сервер на Python, который реализует архитектуру Рекурсивных Языковых Моделей (Recursive Language Models, RLM) в гибридном облачно-локальном исполнении. 

Цель системы — позволить мне (пользователю) делегировать сложные задачи тебе (Облачной Root-модели), при этом не загружая весь объем локальной памяти проекта в твое контекстное окно. Вместо этого ты будешь работать в изолированной Python-среде (REPL), писать скрипты для фрагментации локальных файлов памяти и программно запускать локальную LLM (Qwen) для рекурсивного поиска, фильтрации и извлечения данных, после чего аккумулировать результаты и обновлять локальную память.

## 2. Научное обоснование и проблематика (Основано на исследованиях RLM)
Эта архитектура решает фундаментальные проблемы современных LLM при работе с огромными контекстами (Inference-time scaling) [1]:
*   **Context Rot (Гниение контекста):** При загрузке длинных текстов в промпт качество рассуждений модели резко падает [2].
*   **Сжатие данных:** Стандартные системы памяти (RAG или суммаризация) теряют мелкие, но критичные детали при сжатии [3].
*   **Решение RLM:** Длинный промпт (память) рассматривается не как текст, а как *внешняя среда (Environment)* [4]. Root-модель (ты) не читает память глазами. Ты пишешь код для ее анализа, разбиваешь на куски и запускаешь дочерние Sub-LMs (локальную модель), сохраняя свое контекстное окно чистым для глобального планирования [5]. Это позволяет обрабатывать объемы данных вплоть до 10+ миллионов токенов без потери качества [6].

## 3. Архитектура системы и роли
Проект состоит из трех ключевых узлов, которые должен связать MCP-сервер:

1.  **Root LM (Ты — Облачная модель):** Планировщик и программист. Твоя задача — получить мой запрос, написать Python-код для исследования памяти, запустить REPL-цикл и принять финальное решение [7].
2.  **REPL Environment (MCP Сервер):** Изолированная среда выполнения Python-кода, которая держит в оперативной памяти переменные (буферы) и имеет доступ к файловой системе [5].
3.  **Sub-RLM (Локальная модель Qwen 14B):** «Чернорабочий», работающий через API локального инференса (например, vLLM/Ollama). Опрашивается исключительно через функцию `llm_query(prompt, context_chunk)`, которую ты вызываешь в своем коде [8].

## 4. Требования к структуре данных (Файлы памяти)
MCP-сервер должен организовать хранение памяти в виде структурированных текстовых файлов (Markdown или JSONL). 
*   Данные должны быть легко парсимыми с помощью стандартных методов Python (например, регулярные выражения `re.split(r'### (.+)', text)` или построчное чтение `split('\n')`) [9, 10].
*   MCP-сервер должен предоставлять функцию для получения метаданных о памяти (названия файлов, их размер в символах, структура заголовков), чтобы ты (Root LM) знал(а), с чем имеешь дело, не читая содержимое целиком [7, 11].

## 5. Рабочий процесс (Флоу REPL-цикла)
MCP-сервер должен реализовать цикл Read-Eval-Print Loop (REPL), работающий по следующему алгоритму [12]:

1.  **Инициализация среды:** Я отправляю задачу. MCP-сервер подгружает файлы памяти в глобальные строковые переменные внутри Python-среды [8].
2.  **Твой ход (Code Generation):** Ты получаешь метаданные среды и пишешь Python-код. Код передается MCP-серверу в блоках ` ```repl ` [13].
3.  **Выполнение (Execution):** MCP-сервер выполняет твой код. В коде ты можешь:
    *   Использовать `print()` для вывода небольших кусков текста (stdout вернется тебе в следующем сообщении) [8].
    *   Разбивать память на безопасные чанки (по 10 000 – 15 000 символов, так как локальная модель имеет лимит) [14].
    *   Запускать циклы `for chunk in chunks:` и вызывать `answer = llm_query(...)` для передачи чанков локальной модели [13, 15].
    *   Сохранять ответы локальной модели в массивы (буферы): `buffers.append(answer)` [9].
4.  **Сборка:** Этот цикл (шаги 2-3) повторяется, пока ты не соберешь достаточно данных. 
5.  **Завершение и обновление:** Когда данные собраны, ты формируешь финальный ответ с помощью вызова специальной функции `FINAL_VAR(final_answer)` [16]. После этого ты пишешь последний кусок Python-кода для перезаписи/дополнения файлов памяти обновленной информацией (стандартными операциями `with open(..., 'a')`).

## 6. Требования к функционалу MCP-сервера
Разрабатываемый тобой MCP-сервер должен реализовывать следующие инструменты (tools):

*   **`execute_repl_code(code: str)`**: Принимает твой Python-код, выполняет его в сохраняющей состояние (stateful) песочнице и возвращает тебе `stdout` и ошибки.
*   Внутри песочницы глобально должна быть доступна функция **`llm_query(prompt_instruction: str, context_chunk: str)`**: Эта функция под капотом делает HTTP-запрос к локальной модели Qwen.
    *   *Важно для реализации:* Вызовы `llm_query` в циклах могут сильно тормозить процесс. Сервер должен поддерживать асинхронное выполнение запросов к Sub-LM (например, `asyncio.gather`), чтобы ты мог(ла) запускать обработку десятков чанков параллельно [17, 18].
*   Внутри песочницы глобально доступна переменная **`memory_context`** (например, словарь `{"project_rules.md": "текст...", "user_profile.json": "текст..."}`).

## 7. Требования к Промптам для локальной модели (Sub-LM)
Локальные небольшие модели (уровня 14B) склонны сбиваться и требуют жестких инструкций [19]. В MCP-сервере или в твоих скриптах вызов `llm_query` должен строго форматировать задачу для Qwen. 
*   *Пример правильного вызова в твоем коде:* `llm_query("Извлеки только имена и даты, верни в формате ИМЯ | ДАТА. Если нет совпадений, верни пустую строку. Текст: {chunk}")` [20]. 
*   Qwen не должна «думать» глобально, она должна работать как интеллектуальный regex-фильтр [21]. Рекомендуется батчить данные: передавать в один вызов `llm_query` оптимальное количество информации (например, по 3-5 логических блоков), чтобы снизить накладные расходы времени [14, 22].
